## Q1. Why does Helios use filesystem checkpointing for agent state instead of in-memory snapshots?

### Answer
Helios uses **persistent checkpointing through its memory store and session store** instead of plain in-memory snapshots because the system is designed for **long-running autonomous sessions** that may outlive a single process or context window. In-memory snapshots would disappear if the process crashes, restarts, or if the provider context is reset.

Helios explicitly writes durable state such as goals, observations, best results, and context gists into its memory/session persistence, then rebuilds a briefing from that stored state when the context window fills up. This makes recovery possible even after the original conversational history is archived.
