## Q10. Why does Helios make checkpoint boundaries explicit and developer-controlled rather than automatic?

### Answer
Helios makes checkpoint boundaries explicit and developer-controlled so that checkpointing happens at **predictable, tested, semantically meaningful points** rather than at arbitrary runtime states. This protects consistency of tool calls, streaming responses, session accounting, and recovery summaries.

The design also reflects that checkpointing is a policy decision: thresholds depend on model context windows, actual token usage, and memory quality. By encoding checkpoint handling in `ContextGate` and invoking it deliberately from the orchestrator after completed turns, Helios keeps control over what gets persisted and how recovery resumes.
